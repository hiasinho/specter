Specter

A sync layer for your team's documents, built for AI agents and humans alike.

Keep specs, docs, and decisions in sync across repos and teams. Push from your repo, collaborate through agents or the API.

How it works

Specter sits between your repo and your collaborators. You push docs from the CLI, everyone else — humans and agents — works through the API.

01

Point it at your docs

A .specter file tells the CLI which folders to sync. That's the only setup. 

project: acme/api-specs
paths:
  - specs/
  - docs/architecture/
exclude:
  - specs/drafts/
02

Push and pull

The CLI detects your git branch, hashes content for conflict detection, and syncs only what changed. 

$ specter push
Pushed to acme/api-specs/main:
  created: 3
  updated: 2
  unchanged: 7

$ specter history specs/auth.md
  r5  2026-03-12T10:00:00Z  alice
  r4  2026-03-11T14:30:00Z  bob
  r3  2026-03-10T09:15:00Z  alice
03

Agents and humans propose

Anyone with access can read docs and submit proposals — structured changes anchored to content, not line numbers. 

$ specter proposals
2 pending for specs/auth.md

$ specter review abc123 accept
✓ Proposal accepted

# or via the API:
POST /proposals/acme/api-specs
{ "document_path": "specs/auth.md",
  "type": "replace",
  "anchor_content": "## Auth Flow",
  "body": "..." }

Why Specter

Docs stay in your repo

Your markdown files live where your code lives. Specter syncs them — it doesn't own them. Pull anytime to get changes back.

Built for agents

RESTful API with token auth. LLM agents can read your specs, propose changes, and get reviewed — no git access needed.

Conflict detection, not surprises

Content hashing and revision tracking. Push is rejected if someone else changed the same doc. You always know before you overwrite.

Branch-aware by default

Documents follow your git branches automatically. Work on a feature branch, sync its docs separately, merge when ready.

Proposals, not PRs

Lightweight structured feedback — replace, insert, delete, annotate. Anchored to content so they survive edits. Review from CLI or API.

Zero infrastructure

Single binary CLI, hosted service. No servers, no deploy pipelines, no config beyond one YAML file.

The CLI

Sync

specter push — push local docs to the service

specter pull — pull remote changes to local

specter status — show what's changed locally or remotely

specter diff — show differences between local and remote

History

specter history <path> — list revision history

specter history <path> <rev> — show a specific revision

specter history diff <path> — compare two revisions

Collaboration

specter propose — create a proposal for a document

specter proposals — list proposals for the project

specter review — accept or reject a proposal

specter invite create — create an invite for a role

specter invite redeem — redeem an invite code

Account & Projects

specter register — create a new account

specter me — show current user info

specter projects — list your projects

specter init — create a project on the service

specter project delete — delete a project

Status

Specter is under active development. The API and core sync are functional, but things will change.
